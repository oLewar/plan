## Структура каждого инструмента

Каждый инструмент должен содержать строго 5 полей:[](https://www.youtube.com/watch?v=rN4_Y67Tr8I&t=2560s)

text

`Name:        place_coffee_order / get_calendar_events Description: [что делает инструмент] Params:      json Response:    json Errors:      → bool`

---

## Правило 1 — Имя: глагол + существительное

Имя инструмента должно однозначно говорить, что именно он делает. Формат: `глагол_существительное`.

- **Хорошо:** `place_coffee_order`, `get_calendar_events`, `check_budget`
    
- **Плохо:** `order`, `calendar`, `tool1`
    

Модель выбирает инструмент **по имени**. Если имя размытое — агент ошибётся в выборе или вызовет не тот инструмент.[](https://www.youtube.com/watch?v=rN4_Y67Tr8I&t=2560s)

---

## Правило 2 — Description: позитивные инструкции, не запреты

Description — это главный канал коммуникации с LLM. Он должен:

- Объяснять **что** делает инструмент и **когда его нужно вызвать**
    
- Формулироваться позитивно: _«Используй этот инструмент когда...»_, а не _«Не используй этот инструмент если...»_
    
- Быть достаточно конкретным, чтобы модель не гадала
    

---

## Правило 3 — Params: строгая JSON-схема с валидацией

Параметры описываются через строгую схему (JSON Schema). Без строгих схем возникает главная проблема агентского цикла — **неправильные параметры**: агент вызывает тул с неверными данными. Классический пример из видео: `quantity=999` вместо `1`, или `terraform destroy` вместо точечного удаления.[](https://www.youtube.com/watch?v=rN4_Y67Tr8I&t=2560s)

Обязательно указывать:

- типы всех полей
    
- required-поля
    
- допустимые диапазоны/значения там, где это критично
    

---

## Правило 4 — Response: структурированный JSON, не текст

Ответ инструмента должен быть предсказуемым структурированным JSON, который LLM сможет однозначно распарсить и встроить в следующий шаг рассуждения. Текстовые ответы — источник галлюцинаций и ошибок интерпретации.[](https://www.youtube.com/watch?v=rN4_Y67Tr8I&t=2560s)

---

## Правило 5 — Errors: ошибка как инструкция, не как сбой

Поле `Errors → bool` — критически важное правило. Ошибка инструмента **не должна вызывать падение агента**. Она должна возвращаться как **понятная инструкция для следующего шага модели**.[](https://www.youtube.com/watch?v=rN4_Y67Tr8I&t=2560s)

Правильно: `{"success": false, "error": "item_not_in_menu", "suggestion": "try get_menu first"}`

Неправильно: бросить исключение или вернуть HTTP 500 без объяснений.

Агент должен **получить информацию**, что пошло не так, и принять следующее решение сам — это часть agent loop.

---

## Правило 6 — Лимит на количество вызовов одного инструмента

В agent loop обязательно ставить **лимит на число вызовов одного тула подряд**. Если агент вызывает `get_weather` четвёртый раз подряд — это аномалия, погода не изменилась за 3 секунды. Лимит прописывается **в коде**, не в промте — мягкие ограничения в тексте агент может проигнорировать.[](https://www.youtube.com/watch?v=rN4_Y67Tr8I&t=2550s)

---

## Антипаттерн — MCP как автоматическая обёртка

Отдельно критикуется слепое использование MCP для авто-генерации инструментов из API. Автоматические обёртки учитывают только **happy path** — успешный сценарий. Они не описывают, что делать при таймауте, невалидных данных, частичном успехе. Для продакшена каждый инструмент должен быть написан руками с полной обработкой всех edge cases.
